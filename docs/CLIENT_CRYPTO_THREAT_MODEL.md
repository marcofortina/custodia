# Custodia client crypto threat model

This document summarizes the security boundaries for Custodia high-level client crypto. It complements [`CLIENT_CRYPTO_SPEC.md`](CLIENT_CRYPTO_SPEC.md) and the namespace/key workflow model in [`SECRET_KEYSPACE_MODEL.md`](SECRET_KEYSPACE_MODEL.md).

## Security goals

Custodia client crypto is designed so that the server remains a metadata-only control plane:

- clients encrypt plaintext locally before upload;
- clients generate and keep DEKs outside the server;
- clients create recipient envelopes outside the server;
- clients decrypt only after authorization and local envelope opening;
- the server stores ciphertext, metadata and opaque envelopes, but never plaintext, DEKs, private keys or public-key directory state.

The primary confidentiality goal is that compromise of the Custodia server database alone does not reveal plaintext secret values.

## Trust boundaries

| Component | Trusted for | Must not receive |
| --- | --- | --- |
| Application/client | plaintext, DEK generation, local private keys, recipient key resolution | n/a |
| Custodia server | authentication, authorization, storage, audit, metadata routing | plaintext, DEKs, private keys |
| Signer service | mTLS certificate issuance and lifecycle | secret plaintext or DEKs |
| External key source/resolver | recipient public-key authenticity | Custodia server private keys or plaintext |
| Audit/export pipeline | immutable operational evidence | plaintext, envelopes interpreted as secrets |

The server is not a public-key directory. Recipient public keys must be resolved through application configuration, pinned keys, an enterprise KMS/directory, provisioning, or another out-of-band trust channel.

## Protected assets

High-level crypto protects:

- secret values uploaded through SDK/CLI crypto workflows;
- DEKs generated by clients;
- recipient-specific envelope contents;
- local private key material used to open envelopes.

It does not hide:

- `namespace`;
- `key`;
- owner and recipient client ids;
- version counts and timestamps;
- access grant metadata;
- audit and operational metadata.

Use non-sensitive namespace/key names. Treat them as searchable metadata, not encrypted payload.

## AAD binding

Client-side AEAD uses deterministic AAD bound to the public keyspace:

```json
{
  "version": "custodia.client-crypto.v1",
  "content_cipher": "aes-256-gcm",
  "envelope_scheme": "hpke-v1",
  "namespace": "db01",
  "key": "user:sys",
  "secret_version": 1
}
```

This binding prevents ciphertext/envelope metadata from being silently swapped across namespaces, keys, crypto versions, envelope schemes or versions. Clients must fail closed when AAD cannot be reconstructed or authentication fails.

secret_id is intentionally not part of client crypto AAD. It is an internal storage/FK/audit correlation identifier, not the public workflow identity.

## Nonce requirements

AES-256-GCM content encryption requires a fresh 96-bit random nonce for every DEK encryption operation. SDKs must generate content nonces internally with a language-native CSPRNG and must not expose production APIs that encourage caller-provided static nonces.

HPKE-v1 envelope encryption may use nonce material derived from the HPKE key schedule. That nonce is not interchangeable with the AES-GCM content nonce.

Nonce reuse with the same AES-GCM key is catastrophic and can compromise confidentiality and integrity. Tests may use deterministic fixtures only under `testdata/client-crypto/v1/`.

## Versioning and updates

Updates create a new secret version. Clients bind the next ciphertext to `secret_version + 1` in AAD. Previous versions remain auditable and independently authenticated.

The server must not rewrite ciphertext, crypto metadata or recipient envelopes to simulate an update. It stores the opaque payload selected by the authorized client.

## Sharing and revocation

Sharing rewraps an existing DEK locally for a target recipient and uploads only the new recipient envelope. The server authorizes and stores the envelope, but does not learn the DEK.

Revocation prevents future server-authorized reads and future envelope delivery for the revoked client. It cannot claw back plaintext, ciphertext or envelopes already downloaded by that client before revocation.

Owner delete semantics are separate from recipient self-removal:

- recipient delete of a shared key removes that recipient visibility;
- owner delete removes the owned secret only when no active shares remain, unless cascade is explicit;
- cascade also removes shares before deleting the owner secret.

## Attacker model

Custodia client crypto should remain safe against:

- passive database compromise without client private keys;
- accidental server-side logging of ciphertext/envelopes being treated as opaque data;
- ciphertext swap attempts across `namespace/key/secret_version`;
- unauthorized recipient reads when the server enforces access grants;
- stale SDKs trying to use unsupported crypto metadata versions.

It does not protect against:

- compromised client host or application process;
- malicious or compromised recipient public-key resolver;
- recipient private-key compromise;
- plaintext already exported by an authorized client;
- weak namespace/key names leaking business meaning through metadata;
- server denial of service or metadata deletion when storage controls fail.

## Operational requirements

Production deployments should enforce:

- mTLS client identity checks;
- least-privilege access grants;
- audit verification and export;
- rate limits on sensitive admin/client routes;
- secure local storage for private keys;
- key rotation procedures for client identities;
- secure backup/restore for metadata without exposing plaintext.

## Regression guardrails

Security review must fail if a public workflow reintroduces:

- `--secret-id` for normal CLI operations;
- `name` as a legacy alias for `key`;
- high-level crypto helpers addressed by `secret_id`;
- caller-provided static content nonces in production encryption APIs;
- server-side plaintext, DEK, private-key or public-key-directory behavior.

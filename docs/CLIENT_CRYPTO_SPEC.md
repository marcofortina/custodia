# Custodia client crypto specification

This document defines the shared client-side crypto contract for Custodia high-level crypto clients. It is implemented by the repository Go, Python, Node.js/TypeScript, Java, C++ and Rust SDKs and remains the contract for future clients.

The server remains metadata/ciphertext/envelope-only. It must never receive plaintext, DEK material, private keys or public-key directory state.

## Versioning

Every encrypted payload must carry versioned crypto metadata:

```json
{
  "version": "custodia.client-crypto.v1",
  "content_cipher": "aes-256-gcm",
  "envelope_scheme": "hpke-v1",
  "content_nonce_b64": "base64url-or-standard-base64-nonce",
  "aad": {
    "secret_id": "optional-stable-secret-id",
    "secret_name": "optional-secret-name",
    "version_id": "optional-version-id"
  }
}
```

Unsupported versions must fail closed with a typed client error.

## Content encryption

Recommended v1 content encryption:

- random 256-bit DEK generated by a language-native CSPRNG;
- AES-256-GCM;
- 96-bit random nonce;
- authentication tag included with ciphertext encoding;
- AAD bound to stable Custodia metadata.

The server stores only the encoded ciphertext and metadata. The v1 metadata may also carry the base64 content nonce and the canonical AAD binding chosen by the client. The nonce is not secret, but it is required to decrypt AES-GCM ciphertext.

## Envelope encryption

Recommended v1 envelope scheme:

```text
envelope_scheme = hpke-v1
```

Each recipient envelope wraps the DEK for exactly one recipient identity. The recipient public key must be resolved by the application/client through a local or external resolver. Custodia server must not become a public-key directory.

Legacy envelope schemes, if introduced later, must be named by schema/version, not by implementation language.

## Required client interfaces

```text
PublicKeyResolver.resolve(client_id) -> recipient public key
PrivateKeyProvider.current() -> local private key material or decrypter handle
RandomSource -> language-native CSPRNG
Clock -> testability only
```

## Required AAD inputs

High-level clients must bind encryption to stable metadata. At minimum, v1 AAD must include:

- crypto version;
- envelope scheme;
- content cipher;
- secret name or secret id when available;
- version id when creating a new version after the server returns it, or a documented pre-commit AAD value before persistence.

The exact canonical AAD serialization is deterministic JSON produced by `internal/clientcrypto.BuildCanonicalAAD` and covered by fixtures. The current canonical object order is `version`, `content_cipher`, `envelope_scheme`, then optional resource bindings `secret_id`, `secret_name`, `version_id`. High-level clients persist the selected AAD binding in `crypto_metadata.aad` so read/share/version paths can reproduce the same AAD without asking the server to expose extra plaintext metadata.

## Error model

All client languages must expose equivalent errors for:

- unsupported crypto version;
- unsupported envelope scheme;
- missing recipient envelope;
- wrong recipient key;
- AAD mismatch;
- ciphertext authentication failure;
- malformed metadata;
- random source failure.

## Logging rules

Clients must not log:

- plaintext;
- ciphertext;
- envelope;
- DEK;
- private key material;
- passphrases;
- bearer/session material.

Debug logs may include only operation names, HTTP status codes, request ids and sanitized metadata.

## Test vectors

All official crypto clients must pass vectors under:

```text
testdata/client-crypto/v1/
```

The repository ships deterministic v1 crypto fixtures. Each non-schema fixture includes `aad_inputs`, `canonical_aad`, `canonical_aad_sha256` and `expected_error`. Positive vectors also include `plaintext_b64`, `content_dek_b64`, `content_nonce_b64`, AES-256-GCM `ciphertext` and HPKE-v1 recipient `envelope` payloads. Negative vectors cover ciphertext authentication failure, wrong recipient, AAD mismatch and unsupported crypto version.

Fixture private keys and DEKs are test-vector material only. They exist so SDKs can reproduce the same ciphertext/envelope bytes and must never be copied into production clients.


## Repository validator

The repository includes a small metadata validator in `internal/clientcrypto` for the shared v1 constants:

```text
version = custodia.client-crypto.v1
content_cipher = aes-256-gcm
envelope_scheme = hpke-v1
```

This validator also backs the first Go high-level crypto client implementation. It keeps metadata parsing, canonical AAD, AES-256-GCM payload vectors, HPKE-v1 envelope vectors and error names aligned across client implementations.

## 6.4 Bash SDK helper

The Bash SDK helper does not implement client crypto itself. It exposes sourceable shell functions around the `custodia-client` CLI, so encryption, decryption, envelope handling and local key use remain in the Go client command.

Shell workflows must use client JSON profiles that point at local mTLS material and local application crypto keys. Secret material must never be passed as command-line arguments. Use the full language SDKs for application code; use the Bash helper only for ops/CI glue.

# Custodia client crypto specification

This document defines the shared client-side crypto contract for future high-level Custodia crypto clients.

The server remains metadata/ciphertext/envelope-only. It must never receive plaintext, DEK material, private keys or public-key directory state.

## Versioning

Every encrypted payload must carry versioned crypto metadata:

```json
{
  "version": "custodia.client-crypto.v1",
  "content_cipher": "aes-256-gcm",
  "envelope_scheme": "hpke-v1",
  "aad": "custodia-secret-v1"
}
```

Unsupported versions must fail closed with a typed client error.

## Content encryption

Recommended v1 content encryption:

- random 256-bit DEK generated by a language-native CSPRNG;
- AES-256-GCM;
- 96-bit random nonce;
- authentication tag included with ciphertext encoding;
- AAD bound to stable Custodia metadata.

The server stores only the encoded ciphertext and metadata.

## Envelope encryption

Recommended v1 envelope scheme:

```text
envelope_scheme = hpke-v1
```

Each recipient envelope wraps the DEK for exactly one recipient identity. The recipient public key must be resolved by the application/client through a local or external resolver. Custodia server must not become a public-key directory.

Legacy envelope schemes, if introduced later, must be named by schema/version, not by implementation language.

## Required client interfaces

```text
PublicKeyResolver.resolve(client_id) -> recipient public key
PrivateKeyProvider.current() -> local private key material or decrypter handle
RandomSource -> language-native CSPRNG
Clock -> testability only
```

## Required AAD inputs

High-level clients must bind encryption to stable metadata. At minimum, v1 AAD must include:

- crypto version;
- envelope scheme;
- content cipher;
- secret name or secret id when available;
- version id when creating a new version after the server returns it, or a documented pre-commit AAD value before persistence.

The exact canonical AAD serialization is deterministic JSON produced by `internal/clientcrypto.BuildCanonicalAAD` and covered by fixtures. The current canonical object order is `version`, `content_cipher`, `envelope_scheme`, then optional resource bindings `secret_id`, `secret_name`, `version_id`.

## Error model

All client languages must expose equivalent errors for:

- unsupported crypto version;
- unsupported envelope scheme;
- missing recipient envelope;
- wrong recipient key;
- AAD mismatch;
- ciphertext authentication failure;
- malformed metadata;
- random source failure.

## Logging rules

Clients must not log:

- plaintext;
- ciphertext;
- envelope;
- DEK;
- private key material;
- passphrases;
- bearer/session material.

Debug logs may include only operation names, HTTP status codes, request ids and sanitized metadata.

## Test vectors

All official crypto clients must pass vectors under:

```text
testdata/client-crypto/v1/
```

The repository currently ships deterministic metadata/AAD fixtures. Each non-schema fixture includes `aad_inputs`, `canonical_aad`, `canonical_aad_sha256` and `expected_error`. These are not full ciphertext/envelope cryptographic vectors yet; language clients must not be declared high-level crypto clients until deterministic ciphertext/envelope vectors are present and passing.


## Repository validator

The repository includes a small metadata validator in `internal/clientcrypto` for the shared v1 constants:

```text
version = custodia.client-crypto.v1
content_cipher = aes-256-gcm
envelope_scheme = hpke-v1
```

This validator is not a high-level crypto client. It keeps metadata parsing, canonical AAD, error names and deterministic-AAD fixtures aligned before full deterministic cryptographic vectors are added.
